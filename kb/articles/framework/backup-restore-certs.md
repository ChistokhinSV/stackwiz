# Backup and restore TLS certificates

Reinstalling a stackwiz VM without touching its certificate
material means the new box can keep serving the same hostnames to
the same clients — no new self-signed CA root, no Let's Encrypt
rate-limit hit, no "untrusted certificate" warnings on every
browser that already trusts the old one.

The `./bootstrap.sh backup-cert` / `restore-cert` subcommands
handle the cert tier end-to-end. Everything else (Vault data,
Consul KV, Authentik DB, docker volumes, component state) is
**out of scope** here and covered by a separate
`backup`/`restore` pair (TBD).

## What gets captured

Two host paths. Everything else is derived and regenerated by
install scripts on the next run.

| Path                  | Contents                                                               |
| --------------------- | ---------------------------------------------------------------------- |
| `/etc/stackwiz/tls/`  | Self-signed CA root (`stackwiz-ca.{crt,key}`), per-host leaves (`<host>.{crt,key,fullchain.crt}`), operator BYOC overrides (`custom/<host>/…`). |
| `/etc/letsencrypt/`   | LE account, `live/<host>/` symlinks, `archive/<host>/` actual cert + private keys, `renewal/*.conf`, `renewal-hooks/`, credential files (`.stackwiz-cloudflare.ini`, `.stackwiz-route53.env`). |

Explicitly **not** in the tarball (regenerated from the above on
the next `./bootstrap.sh run`):

- `/opt/stackwiz/vault/tls/` — copied from `/etc/stackwiz/tls/` by `081/install/vault.sh`.
- `/opt/stackwiz/nginx/tls/` — copied by the nginx install scripts.
- `/var/lib/stackwiz/shared/vault-ca.crt` — published for sibling consumers.

## Backup

From any consumer dir (061, 077, 081, 082, 080…):

```bash
./bootstrap.sh backup-cert            # writes ./stackwiz-certs-<host>-<utc>.tar.gz
./bootstrap.sh backup-cert /tmp       # or into an explicit dir
./bootstrap.sh inspect-cert <tarball> # peek without restoring
```

The bootstrap fast-paths cert commands — it skips Consul/Vault
discovery and the normal install plumbing. The only host
prerequisites are `sudo`, `docker`, and the ability to pull
`ghcr.io/chistokhinsv/stackwiz:latest`.

The tarball is written `0600` because it carries the CA private
key and Let's Encrypt private keys. Always encrypt before moving
off-host:

```bash
gpg --symmetric --cipher-algo AES256 stackwiz-certs-*.tar.gz
# enter a strong passphrase
scp stackwiz-certs-*.tar.gz.gpg ...
# on the copy source VM, after confirming the encrypted copy is
# safe, shred the plaintext:
shred -u stackwiz-certs-*.tar.gz
```

## Restore

On the **new** VM, **before** you run any install step:

```bash
# Bring the encrypted copy in, decrypt:
scp stackwiz-certs-*.tar.gz.gpg newvm:/tmp/
ssh newvm
gpg -d /tmp/stackwiz-certs-*.tar.gz.gpg > /tmp/certs.tar.gz

# Install base packages and clone one consumer repo (anything with
# the bootstrap.sh + stackwiz-bootstrap.sh pair).
cd ~/awx-platform     # example

# Optional — peek:
./bootstrap.sh inspect-cert /tmp/certs.tar.gz

# Restore (refuses to overwrite existing dirs — use --force to swap):
./bootstrap.sh restore-cert /tmp/certs.tar.gz
./bootstrap.sh restore-cert --force /tmp/certs.tar.gz

# Shred the plaintext:
shred -u /tmp/certs.tar.gz
```

Now run installs as usual:

```bash
./bootstrap.sh run --auto
```

Install scripts that touch TLS (`stackwiz-tls.sh`, `vault.sh`,
`kb-nginx.sh`, …) see the restored certs, verify their validity
with the built-in 30-day freshness check, **reuse** them instead
of issuing new ones, and re-populate the derived copies
(`/opt/stackwiz/vault/tls`, `/opt/stackwiz/nginx/tls`,
`/var/lib/stackwiz/shared/vault-ca.crt`).

## `--force` behaviour and the `.before-restore-*` naming

Restore refuses to clobber existing cert dirs by default. With
`--force`, each existing target is **moved aside first** so the
operator can roll back if the restore was a mistake.

Two different "aside" layouts depending on whether the target is
a bind-mount root (always true for the restore-cert docker
context — `/etc/letsencrypt` is bind-mounted 1:1) or a regular
subdirectory of a mounted parent (`/etc/stackwiz` is the mount,
`tls/` is a regular subdir inside):

```
/etc/stackwiz/tls.before-restore-20260421-104406   # sibling rename (regular dir)
/etc/letsencrypt/.before-restore-20260421-104406   # subdir inside the mount (dot-prefixed)
```

Both preserve the original contents; they're safe to delete
after you've verified the restore worked:

```bash
sudo rm -rf /etc/stackwiz/tls.before-restore-*
sudo rm -rf /etc/letsencrypt/.before-restore-*
```

## Quick-verify after restore

```bash
# CA root + leaf present:
sudo ls -la /etc/stackwiz/tls/ | head -6

# LE certs intact, not-before / not-after still in the expected range:
sudo openssl x509 -in /etc/letsencrypt/live/<host>/fullchain.pem \
  -noout -dates -subject

# Browser / client trust chain — certificate fingerprint unchanged:
echo | openssl s_client -connect <host>:443 -servername <host> 2>/dev/null \
  | openssl x509 -noout -fingerprint -sha256
```

Compare the fingerprint with what clients already trust. If it
matches, the reinstall is invisible to every existing client.

## Common pitfalls

**"no cert paths on this host"** — the vendored
`stackwiz-bootstrap.sh` on disk is older than the image and the
cert-fast-path isn't active. Fix:

```bash
docker pull ghcr.io/chistokhinsv/stackwiz:latest
# The bootstrap now auto-refreshes its library on every run, so a
# second invocation should work. If it still doesn't, force a
# manual re-vendor:
docker run --rm -v "$PWD:/out" ghcr.io/chistokhinsv/stackwiz:latest \
  extract-bootstrap --output /out --force
```

**Partial restore** — if `restore-cert --force` crashes
mid-operation (e.g. disk full), the already-restored path is in
its new state, the rest is untouched. Re-running with the same
tarball resumes from where it stopped (already-existing dirs
trigger the `--force` move-aside again, so you end up with
multiple `.before-restore-<stamp>` siblings — noisy but safe).

**Hostname changed** — certs are issued for specific FQDNs. A
restored cert works only if the new VM serves the same hostname.
If you rename the box, let `stackwiz-tls.sh` issue a fresh leaf
(the CA root still survives — existing clients keep trusting it).

**Docker + postgres version mismatch** (out of scope for
`backup-cert`, but worth mentioning for full-DR planning):
Authentik / Wiki.js / MCPJungle use named volumes with specific
Postgres versions. Raw volume copies fail across major-version
upgrades. Use per-service `pg_dump` / `pg_restore` when upgrading.

## Where the logic lives

- Shell fast-path: `src/stackwiz/share/bootstrap/stackwiz-bootstrap.sh`
  (`sw_cert_fast_path` — resolves paths, bind-mounts `/etc/stackwiz`,
  `/etc/letsencrypt`, `/cert-in`, `/cert-out`).
- Python dispatcher: `src/stackwiz/cli.py` `backup-cert` / `restore-cert` /
  `inspect-cert` click subcommands.
- Core logic: `src/stackwiz/cert_backup.py` — `backup()`, `restore()`,
  `inspect()`; all stdlib (tar + shutil), no extra deps.
